## HomeAssistant 重启死循环 + HASS.Agent 认证修复

### HA 权限问题
- 之前 `user: 1000:1000` 不适用于 HA 容器——s6 init 阶段以 root 创建文件，主进程降权后无法访问，导致 PermissionError 死循环重启
- 修复: 去掉 `user:` 指令，HA 内部自行管理权限
- 期间临时以 root 启动容器，`docker exec` chown 修复了大量 root 属主文件

### HASS.Agent 认证失败
- `192.168.1.160` 上 HASS.Agent 用旧 HA 实例的凭据连接全新 HA，每 5 秒报 Login attempt failed
- 解决办法:
  1. HA 侧: 用户头像 → 安全 → 长期访问令牌 → 创建 `HASS-Agent-Huang`
  2. HASS.Agent 侧: Server URI 改为 `http://192.168.1.146:8123`（不用 homeassistant.local）
  3. 重要: 先点保存再点测试，否则配置不持久化

## 备份方案最终化

- 确定备份/还原统一方案: **`sudo ds-backup`** — 一次密码，全流程以 root 运行
- 宿主机直接 tar 遇到权限问题（HA `.storage/` 混有 root/1000 属主，mosquitto `.db` 属主 1883）
- `sudo tar` 完整保留 UID/GID，还原后权限一致
- 备选方案 `docker run -v alpine tar` 同样有效但需在 docker 组
- 更新了 `backup.sh` 注释和 MEMORY.md，推荐 sudo 执行
- `--install` 改到 `/usr/local/bin/`（sudo PATH 可用），不再回退 `~/.local/bin`
- `main()` 强制要求 root：非 root 执行备份/还原直接拒绝，提示用 sudo
- 例外: `--install` / `--uninstall` / `--help` 不需要 root

## Docker 镜像迁移功能

- 新增 `sudo ds-backup export-images`: 导出已部署容器的镜像到 `images/`，仅导出实际运行的容器（通过 container_name 匹配 docker ps -a）
- 新增 `sudo ds-backup load-images`: 从 `images/*.tar` → `docker load` 到本地 Docker
- images/ 加入 .gitignore，不纳入版本管理
- Tag 漂移保护: 导出时对比容器实际 Image ID 与 compose tag 当前指向的 ID，不一致时保存容器版本 + `.tag` 侧车文件，load 时自动重新打标
- docker group 中的 fishme 用户已移除（安全问题），当前 session 缓存可能未清除


## Docker 镜像迁移功能（已彻底移除）

原因: Docker 29 + containerd snapshotter (`io.containerd.snapshotter.v1`) 下 `docker save` 存在 bug，HA 等部分镜像导出只有 manifest/config（80K）无实际 layer 数据。已彻底移除所有镜像导出/加载功能，迁移时直接 `docker compose pull` 即可。
