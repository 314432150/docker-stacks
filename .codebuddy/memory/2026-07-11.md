## fetchWithError: DELETE 报 "Unexpected end of JSON input" 修复

**根因**: `fetchWithError` 对无 body 的请求（GET/DELETE）也设置了 `Content-Type: application/json`，Fastify 自定义 body parser 对空 body 调用 `JSON.parse("")` → 500。

**修复**: `useSSE.js` — 仅在 `body` 存在时才设 `Content-Type` 头。GET 不受影响（Fastify 对 GET 不触发 body parse）。

## 白屏 + 重复鉴权修复

**根因 1（白屏）**: `app.mount('#app')` 先于 `router.beforeEach` 的异步鉴权完成 → App 渲染时路由未就绪 → 短暂白屏。

**根因 2（重复鉴权）**: `main.js` guard 和 `App.vue checkAuth()` 各自独立调用 `/api/auth/status`，全局 2 次/页。
**修复**: 创建共享 `authStore`（reactive），Guard 写入 isAuthenticated/setupNeeded/currentUser，App.vue 直接读取。删除 App.vue 的 `checkAuth()` 函数，减少为 1 次/页。

**apps 拉取触发规则**:

- Backup/Deploy: `onMounted/onActivated` → `fetchApps()`（30s 缓存）
- Restore: `onMounted/onActivated` → `fetchApps()` + `fetchBackups()`（30s 缓存 + 无缓存）

- 30s 内重复请求合并为一个实际 HTTP 请求（`withCache` 去重）

## 循环引用致白屏 + 重复鉴权（第二轮修复）

上次修复在 `main.js` 内定义 `authStore` 并让 `App.vue` import，引入了 **循环引用**：
`main.js` → `import App from './App.vue'` → App.vue → `import { authStore } from './main.js'`

ESM 导入提升导致 `authStore` 初始化先于 `export const authStore = reactive(...)` 执行，生产构建 (Rollup) 可能产出两个不同的 authStore 引用：
- 后果 1：Guard 写入的 `checked=true` 落在一个 authStore，App.vue 读的是另一个（永远 false）→ 每次导航都鉴权
- 后果 2：authStore 属性为 undefined → `if (authStore.setupNeeded)` 可能抛异常 → `router.isReady()` 永远不 resolve → `app.mount` 不执行 → 白屏

**修复**：
1. **抽离 `composables/useAuth.js`** — authStore 单文件，无循环依赖，`main.js` 和 `App.vue` 各自独立 import
2. **NMenu 改用编程式导航** — 移除 RouterLink（`h(RouterLink, {to:'/'})`），改为 `@update:value="handleMenuUpdate"` + `router.push(key)`
   - 好处：NMenu 不再创建 RouterLink VNode，避免重复导航事件；`:value="activeMenuKey"` 正确跟踪当前路由高亮
3. **`router.isReady().then(() => app.mount('#app'))` 保留** — 配合 authStore 修复后应正常工作

## 预初始化鉴权 + KeepAlive 双重调用修复（第三轮）

### 3 次 auth/status + 白屏根因

`router.beforeEach` 的 async guard 在 `app.use(router)` 导致的初始导航中执行，与 `router.isReady()` 之间存在时序竞争：async guard 抛异常或阻塞 → `isReady()` 永不 resolve → `app.mount` 不执行 → 白屏。同时 Router 初始化 + RouterView 挂载可能触发多次导航 → guard 多次执行。

**修复**: **预初始化模式**
```
checkAuth()  ← 唯一 auth/status 调用
   ↓ (authStore 已填充)
createApp + app.use(router)
   ↓ (beforeEach 同步读取 authStore，isReady 立即 resolve)
app.mount('#app')
```
登录后重检：guest guard 设 `checked=false`，beforeEach 仅在 `!checked` 时异步 `await checkAuth()`（仅从登录页离开时触发，共 2 次/会话）。

### KeepAlive 双重调用

Vue 3 的 KeepAlive 首次渲染同时触发 `onMounted` + `onActivated`。Dashboard/Backup/Restore/Deploy 都注册了两个钩子 → 数据拉取两次。`fetchBackups()` 无缓存 → 肉眼可见双重 HTTP 请求。

**修复**: 移除所有 KeepAlive 组件的 `onMounted`，只保留 `onActivated`（涵盖首次挂载和后续激活）。Dashboard 的 `fetchApps()` 有 30s 缓存，双重调用只命中缓存不发起网络请求，但为一致性也一并修正。

## Settings.vue v-show 致白屏（第四轮）

## checkAuth Promise 去重（第六轮）

**问题**: 用户反馈首次刷新 `/` 仍发起 2 次 `/api/auth/status`。

**根因**: `main.js` 的 `checkAuth()` 和 `Login.vue` 的 `onMounted` 各自独立 `fetch('/api/auth/status')`，两个调用点可能在并发/竞态场景下重叠。

**修复**:
1. **`checkAuth()` 抽到 `composables/useAuth.js`** — 加 `Promise` 去重锁：`checkAuthPromise` 变量确保同一时刻多次调用只发一次 HTTP 请求
2. **`Login.vue` 复用 `checkAuth()`** — `onMounted` 不再直接 `fetch`，改为 `if (!authStore.checked) await checkAuth()`，然后读取 `authStore` 状态
3. **`Login.vue` `doSubmit()` 成功后更新 `authStore`** — `authStore.isAuthenticated = true; authStore.currentUser = u`，否则 `beforeEach` 拦截跳转重定向回 `/login`

**验证**: Playwright 拦截计数 — 首次加载 1 次，刷新 1 次。E2E 46/46 通过。

## /login 页 2 次 auth/status + setup 后不跳转（第七轮）

**问题 1**: 访问 `/login`（needsSetup 状态）发起 2 次 `/api/auth/status`。
- 用户看到 2 次
- 根因: `main.js` 的 `beforeEach` 进入 guest 路由时执行 `authStore.checked = false`（旧逻辑是"离开登录页时需重检"），导致 `Login.vue` `onMounted` 中 `if (!authStore.checked) await checkAuth()` 立即再发请求
- 修复: `main.js` 仅在 `authStore.isAuthenticated` 为 true 时才重置 `checked`（登出场景）；普通进入 `/login` 不重置

- 修复: `doSubmit()` 成功后同步 `authStore.checked=true, isAuthenticated=true, setupNeeded=false, currentUser=u, isSetup.value=false`

**验证**: Playwright — `/login` 页 1 次 auth/status，setup 成功跳转 `/#/`。E2E 46/46 + 后端 66/66 全通过。
